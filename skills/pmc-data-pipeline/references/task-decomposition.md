# PMC 任务拆解方法论

## 核心原则

**按业务阶段拆解，不按文件维度拆解。**

❌ 错误拆法（按文件）：
- 子代理 A：审计 DuckDB Schema + 审计 Python 脚本
- 子代理 B：审计 Skills 文件
- 子代理 C：审计配置文件
→ 各子代理在同一个文件上交叉重叠，产出互相冲突。

✅ 正确拆法（按业务阶段）：
- 子代理 A：**业务全景** — 10 个场景的组装逻辑、数据地图、背景知识缺口
- 子代理 B：**数据接入** — 从标准数据地图出发，引导客户完成接入
- 子代理 C：**解耦审计** — 排查现有 Skill 中的客户耦合（路径、凭证、品牌名）

每个子代理面对的是**不同的业务问题**，产出物之间有清晰的接口（A 的数据地图 → B 的接入向导输入），不存在文件交叉。

## 三类 PMC 调研任务的标准拆法

### 阶段一：业务全景梳理
- 目标：搞清楚哪些场景是串联的（如场景01→02），各自消费什么数据
- 方法：读 Skill 文档 + 查 DuckDB Schema + 对照 DWD 刷新脚本
- 产出：DATA_MAP.md（数据地图）+ 背景知识缺口清单

### 阶段二：数据接入层设计
- 目标：设计一个 Skill，让 PMC 数字人能引导客户完成数据对接
- 方法：基于阶段一的 DATA_MAP 提取标准数据格式，设计对话流程
- 产出：pmc-data-onboarding Skill 草稿

### 阶段三：现有资产解耦审计
- 目标：排查所有 Skill 中的客户耦合（硬编码路径、凭证、品牌名）
- 方法：全文搜索关键模式（`/Users/xxx`、`cosboard`、`8.134.*` 等）
- 产出：审计报告 + 按 P0/P1/P2 分级的修改清单

## 反模式

- **按脚本文件分任务**：子代理 A 审计 Python、子代理 B 审计 SQL —— 重合点在同一个 Skill 里，产出冲突
- **按 Skill 编号分任务**：pmc-00~03 给 A、pmc-04~06 给 B —— 漏掉了公共问题（如所有 Skill 都用同一个硬编码路径）
- **按工具类型分任务**：DuckDB 给 A、Python 给 B、Markdown 给 C —— 同一个人既要看数据又要看代码，数据语义和代码逻辑割裂
